<objective>
Close the one verification gap left open in Phase 1 (01-VERIFICATION.md, WR-02 in 01-REVIEW.md): `discoverTouchedFiles` currently scans a fixed 400-line transcript tail with no turn-boundary detection, so a client file edited many turns earlier — and since deleted, renamed, or reverted — can still cause an unrelated later Stop/SubagentStop event to block with a confusing "arquivo inacessível" message unrelated to the triggering turn's actual work. This plan restores the D-06 "current turn only" guarantee that 01-02-PLAN.md's own must_haves already promised but the implementation never delivered, and adds a defense-in-depth fix so a discovered-but-since-deleted file never blocks even in the current turn.

Purpose: XMLV-05 ("hook runs automatically... without disrupting the pipeline") is only partially satisfied while this gap is open — a genuinely working validator is undermined by spurious, unrelated blocks in normal day-to-day use. This plan makes the "runs automatically without disrupting the pipeline" promise actually true.
Output: No new artifacts — modifies `.claude/hooks/validate-xml-casca.js` and its test suite (`.claude/hooks/validate-xml-casca.test.js`) in place. No new files are created.
</objective>

  <action>
    In `discoverTouchedFiles(transcriptPath, tailLines = 400)`, keep the existing `allLines.slice(-tailLines)` tail-window cap unchanged, but add a private (unexported) helper — e.g. `isGenuineUserTurnStart(parsed)` — placed just above the function: return `false` unless `parsed && parsed.type === "user"`; if `parsed.message` and `typeof parsed.message.content === "string"`, return `true`; if `parsed.message.content` is an array, return `true` only when at least one block has `type !== "tool_result"` (i.e. the message is not composed purely of tool_result relays); otherwise `false`. This distinguishes a genuine human/slash-command turn start from Claude Code's synthetic tool_result-carrier messages — empirically confirmed against this repo's own real transcripts (`/root/.claude/projects/-root-EiPrompt/*.jsonl`) that the large majority of `"type":"user"` lines are pure tool_result relays, not new turns.

    Before the existing per-line loop that extracts Edit/Write tool_use blocks, scan the tail-windowed `lines` array from its last index backward — tolerating blank lines and JSON.parse failures exactly like the existing loop already does — to find the highest index where `isGenuineUserTurnStart` is true; call it `turnStart`. If no such index exists anywhere in the window, set `turnStart = 0` (fail open to the previous whole-window behavior — never narrower than before, so detection is never silently disabled just because a boundary couldn't be identified). Restrict the existing extraction loop to `lines.slice(turnStart)` instead of the full `lines` array. Do NOT key the boundary off any `"type":"user"` line unconditionally: a naive "last user line" cutoff would exclude edits made earlier in the very same turn whenever a tool_result relay follows them before the next real message — the normal tool_use → tool_result → tool_use pattern within one turn — which would regress the hook to detecting almost nothing, a worse outcome than the WR-02 bug being fixed. The genuine-vs-relay distinction above is the load-bearing part of this fix; preserve it exactly.

    In `validateFile(filePath)`: in the existing `catch (err)` branch around `fs.statSync(filePath)`, add a `code: err.code` field to the pushed error object alongside the existing `message: "arquivo inacessível: ${filePath}"`, so callers can distinguish "file no longer exists" (`err.code === "ENOENT"`) from other stat failures (permissions, etc.) without parsing the Portuguese message text. Additionally, wrap the currently-unguarded `const content = fs.readFileSync(filePath, "utf8");` call in its own try/catch, returning the identical `{ valid: false, errors: [{ message: "arquivo inacessível: ${filePath}", code: err.code }] }` shape on any read failure — this closes a pre-existing TOCTOU gap where a file deleted between the statSync check and the readFileSync call would throw uncaught and crash the Node CLI process instead of degrading gracefully (see threat T-1-07 below), which matters more now that runCli's aggregation logic (next paragraph) depends on "file no longer exists" being a well-behaved, non-throwing outcome from validateFile.

    In `runCli(argv)`: where it currently loops `for (const filePath of files) { const result = validateFile(filePath); errors.push(...result.errors); }`, change this so that when `result.valid === false` AND every entry in `result.errors` has `code === "ENOENT"`, that file is skipped entirely — no push into the aggregate `errors` array, no contribution to the eventual block decision — rather than being treated as a violation. Do not apply this skip to any other error shape (a directory path's "não é um arquivo regular", or any genuine casca-structure violation with `line`/`col`) — those must continue to block exactly as before, per D-07 ("the hook always blocks on a broken casca"), which this change does not weaken: it only recognizes that a file with no bytes to read has no casca that can be broken.
  </action>
  <acceptance_criteria>

    - `discoverTouchedFiles` restricts its Edit/Write extraction to lines at or after the most recent genuine turn-start line found in the tail window, falling back to the full window when no such line is found.
    - `validateFile`'s two failure branches (statSync catch, readFileSync catch) both return `errors[0].code` set to the underlying `err.code` (e.g. `"ENOENT"`), and neither branch can throw uncaught.
    - `runCli` omits any discovered file from its aggregate `errors`/block decision when that file's `validateFile` result consists entirely of `ENOENT`-coded errors, while still including any other error shape.
  </acceptance_criteria>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|--------------|
| Transcript JSONL content -> `discoverTouchedFiles`'s new turn-boundary scan | The same trust boundary as Plan 02 (transcript content is produced by a live Claude Code session and is normally trustworthy), now additionally load-bearing for deciding WHICH lines are "in scope" for discovery, not just which lines contain candidate file paths |
| Filesystem existence state -> `validateFile` (statSync AND readFileSync) | A file's existence can change between the moment `discoverTouchedFiles` lists it and the moment `validateFile` reads it (TOCTOU) — this plan's ENOENT handling must degrade gracefully at both check points, not just the first |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-1-06 | Tampering | `discoverTouchedFiles`'s `isGenuineUserTurnStart` boundary detection | low | accept | A transcript line crafted to masquerade as a genuine user-turn boundary could shift the current-turn cutoff and hide an earlier bad edit from validation; this requires the attacker to already have write access to the Claude Code session's own transcript file — a stronger position than this local dev-tool hook is designed to defend against, the same trust level already accepted for `.claude/settings.json` tampering in 01-03-PLAN.md's T-1-05. No new mitigation beyond the existing per-line `JSON.parse` try/catch. |
| T-1-07 | Tampering | `validateFile(filePath)`'s `fs.readFileSync` call (TOCTOU: file deleted between the `statSync` check and the `readFileSync` read) | medium | mitigate | Task 1 wraps the previously-unguarded `fs.readFileSync` call in its own try/catch, returning the same `{valid:false, errors:[{message:"arquivo inacessível: ...", code: err.code}]}` shape as the existing `statSync` catch branch on any read failure — a race-deleted file now degrades gracefully instead of throwing uncaught and crashing the Node CLI process. |
| T-1-08 | Denial of Service | `discoverTouchedFiles`'s new backward boundary-scan loop over the tail window | low | accept | Bounded by the pre-existing `tailLines = 400` cap (unchanged by this plan); the added backward scan is a single additional linear pass over the same already-bounded window — no new unbounded-input risk is introduced. |

</threat_model>
