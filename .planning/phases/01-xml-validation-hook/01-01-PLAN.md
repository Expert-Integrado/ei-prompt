  <action>
    Implement `.claude/hooks/validate-xml-casca.js` as a CommonJS module (`#!/usr/bin/env node` header, `const fs = require("fs"); const path = require("path");` imports, matching bin/cli.js style exactly):
    - `TIPO_MAP`: const object keyed by exact basename — `"Orquestrador.md":{tipo:"orchestrator"}`, `"Qualifier.md":{tipo:"qualifier"}`, `"Scheduler.md":{tipo:"scheduler"}`, `"Protractor.md":{tipo:"protractor"}`, `"Follow-Up.md":{tipo:"followup"}`, `"Recepcionista.md":{tipo:"orchestrator",origem:"recepcionista"}`.
    - `normalizeContent(content)`: replaces every CRLF pair with a bare LF (CRLF-safe splitting, Pitfall 3) — does NOT strip a leading BOM (BOM is reported, never silently stripped, per D-07/Pitfall 2).
    - `offsetToLineCol(content, offset)`: walks `content` up to `offset` counting newline occurrences for the 1-based line number, and computes the 1-based column as the distance from the last newline before `offset`.
    - `parseAgenteLine(line)`: matches the whole line against an anchored pattern requiring it to open with the `agente` tag name and close with `>` at end-of-line, capturing everything between the tag name and the closing `>` as the attribute string; if no match, return `{ok:false, reason:"linha 2 não é uma tag agente de linha única"}`; else extract EACH attribute individually (order-tolerant, per D-04 — attributes are validated one-by-one for presence/correctness, never matched as a single literal full-line string, so harmless reordering never produces a false positive) via a bounded `name="value"` regex over the captured group into an `attrs` object, return `{ok:true, attrs}`. Both regexes use only bounded, non-nested character classes (T-1-01 ReDoS mitigation) — do not introduce nested/overlapping quantifiers.
    - `countAgenteTags(content)`: an "open" regex matching the literal 6-character tag-name prefix `agente` immediately preceded by `<` and immediately followed by whitespace-or-`>` (a lookahead, NOT a literal match on `>` — since a real open tag never has `>` immediately after the tag name, per Pitfall 1's word-boundary-safety requirement), and a "close" regex matching the literal closing form `</agente>` in full (this closing form is safe as a literal match since it always has `>` immediately after, per Pitfall 1's verified finding); returns `{opens, closes}` as match counts only.
    - `validateCasca(content, basename)`: orchestrates all checks, NEVER throws, returns `{valid: boolean, errors: [{line, col, message}]}`, aggregating every applicable error (does not stop at first failure):
      1. BOM check: if the content's first character code is `0xFEFF`, push `{line:1, col:1, message:"arquivo contém BOM (U+FEFF) antes da declaração XML"}` and drop that leading character for all subsequent checks in this call.
      2. Normalize via `normalizeContent`, split into `lines` on the bare newline character.
      3. XMLV-01: compare `lines[0]` against the exact declaration string `<?xml version="1.0" encoding="UTF-8"?>`; on mismatch, find the first index where the two strings diverge (or the shorter string's length, if one is a strict prefix of the other) and push `{line:1, col: diffIndex+1, message:"linha 1 não corresponde à declaração XML esperada"}`.
      4. XMLV-02/03: call `parseAgenteLine(lines[1] || "")`. If `ok:false`, push `{line:2, col:1, message: result.reason}`. Else, if `TIPO_MAP` has no entry for `basename`, skip tipo/attribute checks entirely (unrecognized file — nothing to validate against). Else for `xmlns` (must equal `https://expertintegrado.com.br/super-sdr/prompt`), `versao` (must equal `1.0`), `tipo` (must equal `TIPO_MAP[basename].tipo`), and `origem` (required ONLY when `TIPO_MAP[basename].origem` is set, must equal that value): for each attribute that is missing, push an error at line 2 col 1 naming the missing attribute; for each attribute present but with the wrong value, locate that value's index within `lines[1]` for the column and push an error naming expected vs found.
      5. XMLV-04: call `countAgenteTags` on the (BOM-adjusted) content; if `opens !== 1 || closes !== 1`, push one error (using `offsetToLineCol` against the offset of the relevant extra open/close match, falling back to end-of-content if no better offset is available) describing the count found vs the expected 1-and-1.
      6. Global invariant: no error `message` string produced by ANY branch may contain the words for "escape" or "CDATA" (in Portuguese or English, any case) — this is not special-cased per-branch, it is simply never part of any message's vocabulary anywhere in this function (D-08).
    - `validateFile(filePath)`: attempt `fs.statSync(filePath)`; on failure (e.g. does not exist), return `{valid:false, errors:[{message: "arquivo inacessível: " + filePath}]}`. If the stat succeeds but `.isFile()` is false, return `{valid:false, errors:[{message: "não é um arquivo regular: " + filePath}]}` (T-1-02 mitigation, mirrors `bin/cli.js`'s `removeFile()` existence/type guard). Otherwise read the file as utf8, call `validateCasca(content, path.basename(filePath))`, and return `{valid: result.valid, errors: result.errors.map(...)}` where each mapped error's `message` is prefixed with the file path and, when `line`/`col` are present, the line/column (format: `<filePath> linha <line> coluna <col>: <message>`; when line/col are absent — the two statSync-failure cases — format as `<filePath>: <message>`).
    - `module.exports = { TIPO_MAP, normalizeContent, offsetToLineCol, parseAgenteLine, countAgenteTags, validateCasca, validateFile };`. Do NOT add `discoverTouchedFiles`, `runCli`, or the `require.main === module` CLI block yet — that is Plan 02's task, and adding it now would create an untested, unreviewed surface.

## Trust Boundaries

| Boundary | Description |
|----------|--------------|
| Filesystem path -> `validateFile` | A file path argument crosses from "believed to be a real client `.md` file" to an actual `fs.readFileSync` call — must be confirmed to be an existing, regular file first |
| Client-authored prose (body content) -> `countAgenteTags` | The body of a real client file is written by a human (not this project's maintainer); this validator's simple boundary-safe regex must not be tricked or crash on adversarial-looking-but-benign content |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|------------------|
| T-1-01 | Denial of Service | `parseAgenteLine` attribute regex, `countAgenteTags` open/close regexes | high | mitigate | Only bounded, non-nested character classes with no adjacent overlapping quantifiers are used in any regex in this file — verified low ReDoS risk by construction (Task 2); Task 3 adds a regression test asserting bounded (<50ms) execution against a 50,000-character crafted attribute value |
| T-1-02 | Tampering | `validateFile(filePath)` | medium | mitigate | Before `fs.readFileSync`, `fs.statSync(filePath).isFile()` is checked (wrapped in try/catch for ENOENT), mirroring `bin/cli.js`'s `removeFile()` path-safety idiom; non-existent/non-regular paths return a structured `{valid:false}` result instead of throwing (Task 2 implementation, Task 3 regression test) |

</threat_model>
