## Phase Goal

**As a** developer maintaining the ei-prompt AI editing pipeline (`docs-editor-conciso`, `client-project-scaffolder`), **I want to** have every client agent file's XML casca checked automatically by deterministic code, **so that** a broken casca is always caught with an actionable file+line/column error — without ever auto-"fixing" the accepted raw `<`/`&` blind spot. This plan specifically delivers the "only validate files actually touched this turn" (D-05/D-06) and "produce the exact block-decision JSON" (XMLV-05/06) slice of that goal.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|--------------|
| Transcript JSONL content -> `discoverTouchedFiles` | Transcript content is produced by a live Claude Code session; while normally trustworthy, a compromised MCP server or prompt-injected tool call could in principle influence a `file_path` value inside a `tool_use` block — this is the point where an "external-ish" input first enters the pipeline |
| CLI argv -> `runCli` | The bash wrapper controls argv (Plan 03), so this boundary is low-risk today, but `runCli` must not crash on malformed/absent argv regardless |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|------------------|
| T-1-02 | Tampering | `discoverTouchedFiles` -> `runCli` -> `validateFile` | medium | mitigate | `discoverTouchedFiles` never itself calls `fs.readFileSync`/`fs.statSync` on a discovered path — it only extracts and basename-filters candidate strings. The actual existence/regular-file check (`fs.statSync(...).isFile()`) lives exclusively in Plan 01's `validateFile`, so every discovered path is safety-checked exactly once, on the one code path that reads file bytes, regardless of how many paths a compromised transcript tries to smuggle in |
| T-1-03 | Denial of Service / Tampering | `discoverTouchedFiles` JSONL parsing | low | mitigate | Every `JSON.parse` call on a transcript line is wrapped in try/catch; a malformed or partial line is skipped, never thrown, bounding the failure to "miss one line's data" rather than crashing the whole hook (Task 1) |

</threat_model>
