## Phase Goal

**As a** developer maintaining the ei-prompt AI editing pipeline (`docs-editor-conciso`, `client-project-scaffolder`), **I want to** have every client agent file's XML casca checked automatically by deterministic code, **so that** a broken casca is always caught with an actionable file+line/column error — without ever auto-"fixing" the accepted raw `<`/`&` blind spot. This plan specifically delivers the "runs automatically on Stop/SubagentStop, coexists with the existing pipeline, ships to end users" slice of that goal (XMLV-05, D-09, D-10).

Add a header comment block, in the same style as `post-ajustes-fanout.sh` lines 1-14, explaining: this hook's purpose (deterministic XML-casca validation on every Stop/SubagentStop cycle), that its output schema is `{"decision":"block","reason":"..."}` for both event types, and — as a deliberate, explained design choice — that this hook re-checks file validity on every single invocation rather than skipping on a retry signal, because its job is a stateless, re-checkable fact rather than a one-shot instruction (D-07/Pitfall 4). Extract `TRANSCRIPT` using the exact idiom from `post-ajustes-fanout.sh` line 30 (`grep -o` for the `"transcript_path"` JSON field followed by `sed` to unwrap the quoted value); guard with `[ -z "$TRANSCRIPT" ] && exit 0`, `[ ! -f "$TRANSCRIPT" ] && exit 0`, `[ ! -r "$TRANSCRIPT" ] && exit 0`. If `$RESULT` is empty or exactly the literal `{}`, print nothing at all to stdout (matching Claude Code's "no output = no block" convention). Otherwise print `$RESULT` verbatim (it is already valid, complete JSON produced by the Node CLI — do not re-wrap, re-parse, or reformat it in bash). Always end the script with `exit 0` regardless of which branch executed (Pitfall 5 — the two existing hooks never exit non-zero, and blocking is communicated purely through the JSON payload on stdout).

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|--------------|
| Hook stdin (Claude Code lifecycle event payload) -> bash wrapper -> `node` subprocess invocation | The wrapper must safely pass an extracted, attacker-influenceable-in-principle string (`$TRANSCRIPT`) into a shell command line |
| `.claude/settings.json` hook registration -> Claude Code hook dispatch | Whoever can write to `.claude/settings.json` can add arbitrary hook commands — this is a pre-existing repo-trust assumption shared by every hook already registered here, not something new introduced by this plan |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|------------------|
| T-1-04 | Tampering | `validate-xml-casca.sh`'s `node ... --transcript "$TRANSCRIPT"` invocation | medium | mitigate | `$TRANSCRIPT` is always referenced double-quoted in every shell use, exactly mirroring `post-ajustes-fanout.sh`'s and `post-scaffolder-review.sh`'s existing idiom — it is never interpolated into an `eval`'d string or used unquoted, so shell-metacharacter injection via a crafted path has no effect (Task 1) |
| T-1-05 | Tampering | `.claude/settings.json` hook registration | low | accept | A compromised `.claude/settings.json` could register arbitrary commands regardless of this phase's changes — this is the same trust level every existing hook in this repo already operates under; out of scope for this phase to redesign |

</threat_model>
