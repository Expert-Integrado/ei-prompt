## Phase Goal

**As a** developer creating a client project via `/ei-cria-cliente`, **I want to** go through three auditable, gated steps (scaffold folders → collect all required fields incl. media → fill templates) with a hard confirmation gate before filling, **so that** no client is ever left with silently incomplete required fields.

<objective>
Update the two pieces of cross-cutting bookkeeping that must change once the 3-step split exists, independent of which mode (single/multi) is used: `manifest.json` must ship the 3 new subagent files and actively retire the old monolithic one for existing installs, and `.claude/hooks/post-scaffolder-review.sh` must retarget its `SubagentStop` audit trigger from the old subagent's name to the new Passo 3 subagent's name (`client-scaffold-fill`), since only the fill step produces content worth auditing.

Purpose: without this plan, Plan 03/04's dispatch of `client-scaffold-fill` would silently produce zero `docs-reviewer` audits (the hook's `case` branch would never match), and end-user `npx` installs would keep the stale monolithic agent alongside the new ones forever (RESEARCH.md Pitfall 3 and Pitfall 4).
Output: `manifest.json` with 3 new `files[]` entries and 1 new `deprecated_files[]` entry; `post-scaffolder-review.sh` with its `case` branch retargeted.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| CLI/manifest distribution boundary | `bin/cli.js` unconditionally trusts `manifest.json`'s `files[]`/`deprecated_files[]` arrays when writing/deleting files in a user's project directory on every `npx` run |
| Transcript-parsing boundary | `post-scaffolder-review.sh` reads an on-disk JSONL transcript file and pattern-matches subagent names from it to decide whether to inject an audit instruction — the transcript content is produced by the same Claude Code session, not an external/untrusted source |

</threat_model>
