## Phase Goal

**As a** developer creating a client project via `/ei-cria-cliente`, **I want to** go through three auditable, gated steps (scaffold folders → collect all required fields incl. media → fill templates) with a hard confirmation gate before filling, **so that** no client is ever left with silently incomplete required fields.

<objective>
Create the three dedicated subagents that replace the monolithic `client-project-scaffolder` (D-01): `client-scaffold-structure` (Passo 1, file I/O only), `client-scaffold-collect` (Passo 2, read-only conversational collector), `client-scaffold-fill` (Passo 3, template writer). Naming these three files is explicitly left to planner discretion by D-02, provided the kebab-case, function-first, shared-prefix convention is followed (satisfied here by the `client-scaffold-*` family, which also keeps `post-scaffolder-review.sh`'s `case` branches greppable as a family, per Plan 02-02) — this plan is where that D-02 discretion is exercised and locked in for every later plan to reference by exact name. Each subagent's `tools:` frontmatter is the structural enforcement boundary for its step — not just a prompt instruction (RESEARCH.md Pattern 1). This plan produces the three files that Plan 03/04 will later chain from `/ei-cria-cliente.md` with a hard confirmation gate in between.

Purpose: without these three purpose-built, tool-scoped subagents, the 3-step split (SCAF-01) has no building blocks — the command file (Plan 03/04) has nothing to dispatch.
Output: three new `.claude/agents/*.md` files, none of which exist yet.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Main command thread ↔ subagent context | Each subagent's declared `tools:` allowlist is the only hard technical boundary between "can ask a question" and "can write a file" — a subagent's prompt text is not a trust boundary by itself, since an LLM can be induced to ignore prose instructions |
| Subagent prompt body ↔ platform capability | A subagent's own prompt cannot grant it a tool it wasn't declared with, but a prompt CAN falsely narrate having done something it structurally cannot do (e.g. claim to have opened a confirmation dialog) |
