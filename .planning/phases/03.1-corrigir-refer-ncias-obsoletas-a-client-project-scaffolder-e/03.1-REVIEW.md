## Summary

Reviewed the 4 files touched by this documentation-drift cleanup phase: the mechanical renames of `client-project-scaffolder` → `client-scaffold-structure` / `client-scaffold-collect` / `client-scaffold-fill` are correct and complete in all 4 files (verified via `grep -c 'client-project-scaffolder'` → 0 in each, and cross-checked against the actual current agent files, `manifest.json`, and `.claude/commands/ei-cria-cliente.md` to confirm the referenced subagents, hooks, and step numbers are real and accurately described). The `inject-ei-context.sh` dual-context CLAUDE.md fallback was functionally tested (both the `client/CLAUDE.md`-present and `client/CLAUDE.md`-absent cases) and behaves as documented.

However, one of the renames did more than rename: in fixing `recepcionista-scaffolder.md`'s Fase 5 confirmation text, the edit replaced an outdated-but-harmless sentence with a new sentence that makes an unconditional completeness claim the agent has no way to verify and that is directly contradicted by the sibling command's own documented behavior (`ei-cria-cliente.md` Passo 4B.1(b)/4B.2). This is a real correctness regression introduced by this phase, not a pre-existing issue, and it cuts against this milestone's own stated Core Value (never let a client scaffold silently look complete when it might not be). See CR-01.

Two lower-severity issues (a documented hook convention not applied to `inject-ei-context.sh`, and two minor completeness/accuracy nits in `.claude/CLAUDE.md`) are also reported below.

## Critical Issues

### CR-01: Fase 5 confirmation text asserts a completeness guarantee the agent cannot verify, and that is false in two real scenarios

**File:** `.claude/agents/recepcionista-scaffolder.md:147`

**Issue:** The Fase 5 confirmation step now reads:

> "Confirme que todas as pastas das especialidades já foram criadas e preenchidas pelo ciclo `client-scaffold-structure` → `client-scaffold-collect` → `client-scaffold-fill` que o `/ei-cria-cliente` executou antes desta etapa (Passo 4B.1(b)) — não há nenhum passo de scaffolding pendente."

This is an unconditional claim of completeness ("já foram... preenchidas", "não há nenhum passo de scaffolding pendente"). It is wrong in two situations that this same agent file and its caller both explicitly support:

1. **Cancellation during the specialty loop.** `.claude/commands/ei-cria-cliente.md` Passo 4B.1(b) (lines 162–164, 203) documents that if the human cancels the Gate de Confirmação for a given specialty, that specialty is registered as **cancelada-e-não-preenchida** — its folder exists but `client-scaffold-fill` was never dispatched, so its templates still contain unfilled `{{variavel}}` placeholders. The loop explicitly **continues** to the next specialty and then on to `recepcionista-scaffolder` regardless (line 200: "a ordem é especialidades → Recepcionista"). So "todas... preenchidas" / "não há nenhum passo pendente" is provably false whenever any specialty was cancelled.
2. **Bypass mode.** Per this same agent's own "ENTRADAS ESPERADAS" section (lines 30–33) and `ei-cria-cliente.md` Passo 4B.2 (line 185: "O loop de especialidades... **não é disparado** neste caminho"), in bypass mode the specialty folders were never touched by `client-scaffold-structure` → `client-scaffold-collect` → `client-scaffold-fill` at all — they "já existem em outro lugar." Yet Fase 5 (lines 143–147) is not scenario-gated, so this same sentence would be presented in bypass mode too, falsely attributing a scaffolding cycle that never ran.

Structurally, the agent cannot even check this: per "ENTRADAS ESPERADAS" (fluxo completo), it only receives a **list of specialty names** (line 27: "Sem `descricao`/`gatilhos`"), with no fill-status field. There is no way for `recepcionista-scaffolder` to know whether a given specialty ended up "preenchida" or "cancelada-e-não-preenchida" — that state only exists in the calling command's own Passo 5 consolidated summary (`ei-cria-cliente.md` line 190).

This directly undermines the milestone's stated Core Value ("`client-scaffold-fill` nunca deve conseguir gerar/deixar um arquivo de cliente com XML quebrado... sem que isso seja pego automaticamente") — telling the operator "nothing is pending" when a specialty's templates may still contain unresolved placeholders is exactly the kind of silent incompleteness this milestone exists to prevent.

**Fix:** Don't assert a completeness state this agent cannot observe. Either drop the claim or make it conditional/deferred to the caller's own tracking:

```markdown
### Fase 5: Confirmação
Apresente ao usuário:
- Caminho dos 4 arquivos criados.
- Lista de campos PENDENTE (se houver).
- Este agente não tem visibilidade sobre o status de preenchimento de cada
  pasta de especialidade (preenchida vs. cancelada-e-não-preenchida) — isso é
  rastreado e reportado no resumo consolidado do Passo 5 de `/ei-cria-cliente`.
  Instrua o usuário a consultar esse resumo para confirmar se alguma
  especialidade ficou pendente de preenchimento.
```

## Warnings

### WR-01: `inject-ei-context.sh` doesn't follow the project's own documented hook convention (`set -uo pipefail`)

**File:** `.claude/hooks/inject-ei-context.sh:1-11`

**Issue:** `.claude/CLAUDE.md`'s "Error Handling" section states as a hard project convention: "`.claude/hooks/*.sh`: use `set -uo pipefail` (not `set -e`...)" — and the other two hooks in the repo (`post-scaffolder-review.sh`, `post-ajustes-fanout.sh`) both follow it. `inject-ei-context.sh` has no `set` directive at all. This phase's plan edited this exact file (adding the `CLIENT_CLAUDE_MD`/`ROOT_CLAUDE_MD` fallback) and was a natural point to bring it in line with the documented convention, but didn't. Currently low risk since the hook is unregistered in `.claude/settings.json` (dead code), but if it is ever re-registered or copy-pasted as a template for a new hook, the missing guard is a latent robustness gap (e.g. a future edit that references an unset variable would silently proceed instead of failing loudly).

**Fix:**
```bash
#!/bin/bash
set -uo pipefail
# inject-ei-context.sh [mode]
...
```

### WR-02: `.claude/CLAUDE.md`'s "Naming Patterns" example list still cites only one of the three post-split subagents

**File:** `.claude/CLAUDE.md:73`

**Issue:** Line 73 lists: "`.claude/agents/docs-editor-conciso.md`, `.claude/agents/docs-analyzer.md`, `.claude/agents/docs-reviewer.md`, `.claude/agents/client-scaffold-fill.md`, `.claude/agents/recepcionista-scaffolder.md`." Before this phase (and before Phase 2's split), this line illustrated the naming convention with the single `client-project-scaffolder.md` example; the rename swapped in `client-scaffold-fill.md` as its replacement but never added `client-scaffold-structure.md` / `client-scaffold-collect.md`, even though those are the other two live files following the exact same naming pattern and are documented two sections later in the Component Responsibilities table (lines 145-147). Not incorrect, but an incomplete update that under-represents the current file set for a reader skimming just this list.

**Fix:** Add the other two, e.g.: `..., .claude/agents/client-scaffold-structure.md`, `.claude/agents/client-scaffold-collect.md`, `.claude/agents/client-scaffold-fill.md`, ...`.

Per `.claude/settings.json`, none of those lifecycle events currently invoke this script (only `SubagentStop`/`Stop` are registered, pointing at other hooks) — this is corroborated by `.claude/CLAUDE.md`'s own v1.8.9 disabled note. **Fix:** Add a short note near the header, e.g. `# NOTE: currently unregistered in .claude/settings.json (disabled since v1.8.9) — see .claude/CLAUDE.md.`
