## Phase Boundary

This phase closes two non-blocking `tech_debt` findings from the v1.0 milestone audit, both caused by the Phase 2 "3-step gated client scaffolding" refactor that retired the single `client-project-scaffolder` subagent (split into `client-scaffold-structure` → `client-scaffold-collect` → `client-scaffold-fill`) without a full sweep for now-stale references, and by Phase 3's dual-context CLAUDE.md fallback pattern not being applied to one already-disabled hook:

1. **Stale `client-project-scaffolder` references** in prose/docs that still describe the retired single-subagent architecture instead of the current 3-step chain.
2. **`inject-ei-context.sh` missing the dual-context (`client/CLAUDE.md`-first) fallback** that every other CLAUDE.md-reading consumer in the repo already has (CLMD-06).

No production/executable wiring is broken by either issue — both are documentation/prose drift or a fallback gap in currently-disabled code. This is a correctness/consistency cleanup phase, not a feature phase.

**Why context was captured this way instead of via `/gsd-discuss-phase`:** the audit doc already pins down file, line numbers, and root cause for both findings with high precision, and the orchestrator's own verification pass (below) confirmed/extended them against the current file contents. There are no subjective design preferences to gather here — every fix target is a factual "what does the code/prose currently say vs. what is actually true" question, fully resolvable by reading the repo. Running an interactive discuss-phase round would mostly re-derive facts already nailed down here.

### Decision 3: `.claude/hooks/inject-ei-context.sh` — add dual-context fallback (CLMD-06 pattern)

Verified: this is the **only** CLAUDE.md consumer in the repo without the fallback (`grep` swept all `.claude/hooks/*.sh`, `.claude/commands/*.md`, `.claude/agents/*.md` — every other consumer already has `client/CLAUDE.md`-first-else-`CLAUDE.md` logic; `check-claude-md-audience.sh` is a different guard, not a context loader, and deliberately excludes `client/CLAUDE.md` from its own scan for unrelated reasons). Verified disabled: `.claude/settings.json`'s `_disabled_note` confirms no `SessionStart`/`UserPromptSubmit`/`PreToolUse` hook currently invokes this script — the fix is latent-risk-only (repo-source-only, per the audit), not an active bug today, but is explicitly named as one of this phase's two targets.

Current (lines 12-13):
```bash
PROJECT_DIR="${CLAUDE_PROJECT_DIR:-$(pwd)}"
CLAUDE_MD="$PROJECT_DIR/CLAUDE.md"
```

Target behavior: prefer `$PROJECT_DIR/client/CLAUDE.md` when it exists, else fall back to `$PROJECT_DIR/CLAUDE.md` (mirrors the exact same dual-context precedence already used by `client-scaffold-fill.md`, `client-scaffold-collect.md`, `client-scaffold-structure.md`, and `.claude/commands/ei-ajustes.md`, just expressed as a bash file-existence check instead of agent-instruction prose since this is a hook script, not an LLM prompt). The line 40 warning ("`[aviso] CLAUDE.md não encontrado em $CLAUDE_MD`") must reflect that both paths were checked, not just one.

Do NOT re-enable the hook (no `SessionStart`/`UserPromptSubmit`/`PreToolUse` registration) — that is out of scope; v1.8.9 disabled it deliberately for unrelated maintenance reasons and re-enabling is not part of this fix.

- Multi-agent bypass: skips the specialty loop entirely, only runs `recepcionista-scaffolder`
- Output: one consolidated final summary (not per-specialty) with per-specialty status (**preenchida** / **cancelada-e-não-preenchida**)
