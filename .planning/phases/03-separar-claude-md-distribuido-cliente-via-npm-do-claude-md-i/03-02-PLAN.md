  <behavior>
    - `normalizeEntry("CLAUDE.md")` returns `{from: "CLAUDE.md", to: "CLAUDE.md"}`
    - `normalizeEntry({from: "client/CLAUDE.md", to: "CLAUDE.md"})` returns the same object unchanged (passthrough)
    - `normalizeEntry({from: "x"})` (missing `to`) throws an Error
    - `normalizeEntry({to: "x"})` (missing `from`) throws an Error
    - `normalizeEntry(null)` / `normalizeEntry(42)` throws an Error
    - `formatManifestEntry("CHANGELOG.md")` returns `"CHANGELOG.md"`
    - `formatManifestEntry({from: "client/CLAUDE.md", to: "CLAUDE.md"})` returns `"CLAUDE.md"` (the `.to` path — what matters to a user reading `--help`, not the repo-source `.from` detail)
    - Neither function's output for an object entry is ever the string `"[object Object]"`
  </behavior>
  <action>
Add two small named functions to `bin/cli.js`, placed near the top alongside `log()` (before `fetchFile`/`writeFile`):

`normalizeEntry(entry)` — if `entry` is a string, return `{from: entry, to: entry}`. If `entry` is a non-null object with both `.from` and `.to` as strings, return it unchanged. Otherwise throw `new Error(...)` with a message that includes `JSON.stringify(entry)` so a bad manifest entry is diagnosable from the logged error (matches RESEARCH.md Pattern 1 exactly).

`formatManifestEntry(entry)` — if `entry` is a string, return it. Otherwise return `entry.to`. This is the single source of truth `help()` uses to render each `files[]`/`deprecated_files[]` line — extract it as its own named function (rather than an inline ternary inside the template literal) specifically so it is unit-testable in isolation without depending on the real `manifest.json`'s current content (RESEARCH.md leaves this exact decomposition to the planner/implementer).

Refactor the install loop (current lines ~93-102, `for (const file of manifest.files)`): rename the loop variable to `rawEntry`, call `const { from, to } = normalizeEntry(rawEntry)` as the FIRST line inside the existing per-file `try` block (so a thrown normalization error is caught by the same `catch` that already handles fetch/write failures — never let it escape and abort the whole batch, matching the project's documented "one failure doesn't abort the batch" convention). Use `from` in the `fetchFile()` call and `to` in the `writeFile()` call. Update the `catch` block's error-logging line to display `typeof rawEntry === "string" ? rawEntry : (rawEntry && rawEntry.to) || JSON.stringify(rawEntry)` instead of the old bare `file` variable, so a failure on an object entry still logs something readable.

Refactor `help()`'s two `manifest.files.map(...)`/`manifest.deprecated_files.map(...)` lines to call `formatManifestEntry(f)` instead of interpolating `f` directly (RESEARCH.md Pitfall 1 — both the loop AND `help()` must be fixed together, since they are physically far apart in the file and it is easy to fix only one).

Finally, wrap the existing bottom-of-file CLI dispatch block (`const cmd = process.argv[2]; if (cmd === "--help"...) ...`) inside `if (require.main === module) { ... }`, and add `module.exports = { normalizeEntry, formatManifestEntry, fetchFile, writeFile, removeFile, run, help };` immediately before that guard — this is the EXACT pattern already established in `.claude/hooks/validate-xml-casca.js` (lines 320-335): requiring the module for tests must never trigger a real install or process.exit. Running `node bin/cli.js <args>` directly must behave 100% identically to before this change.
  </action>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| manifest.json entries → normalizeEntry() → fetchFile()/writeFile() | `normalizeEntry()` is the single gate every manifest entry passes through before touching the filesystem or network; a bug here affects every install |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-03-03 | Denial of Service (self-inflicted) | bin/cli.js install loop | medium | mitigate | `normalizeEntry()`'s throw happens INSIDE the existing per-file `try/catch` (Task 1 action), so one malformed manifest entry logs a `fail` and increments `results.failed` instead of crashing the whole `run()` loop and blocking every other file's install — verified by Task 1's automated check plus the existing per-file resilience convention documented in `.claude/CLAUDE.md` §Error Handling |
| T-03-04 | Tampering | bin/cli.js writeFile() via a compromised manifest `to` field | high | accept | Pre-existing, already-documented risk (T-2-02 in `02-SECURITY.md`): `writeFile()` has no CWD-boundary guard, unlike `removeFile()`. This plan extends the manifest schema but does NOT introduce a new attack surface — `manifest.json` remains a git-controlled file only the maintainer commits, never external/user input, and this plan does not add or change any write-path validation. Risk is reaffirmed as unchanged, not newly introduced or worsened, per RESEARCH.md Security Domain. |
| T-03-05 | Tampering | bin/cli.js help() output | low | mitigate | `formatManifestEntry()` is unit-tested (Task 2) to never render `[object Object]`, preventing a misleading `--help` display that could make a user think an entry silently failed to resolve |
</threat_model>
