01 · O que é

Prompt como código, com processo de engenharia

Os agentes EiPrompt são prompts em Markdown com casca XML, versionados neste repositório. O pacote npm é mínimo — só o CLI e o manifest.json — e todo o conteúdo é baixado da branch main em tempo de execução: atualizou o repo, todo mundo recebe a versão nova sem republicar no npm.

bin/cli.js

Instalador em Node puro, sem dependências. Baixa os arquivos do manifesto, sobrescreve os existentes, remove legados e dá permissão de execução aos hooks.

modelo/*.md

Seis templates de agente com placeholders [VARIAVEL]: Orquestrador, Qualifier, Scheduler, Protractor, Follow-Up e Recepcionista. Pasta read-only — cliente novo é sempre uma cópia.

.claude/

Cinco subagentes, três slash commands e os hooks que automatizam o pipeline de edição dentro do Claude Code.

docs/*.md

As regras do jogo: como editar prompts, checklists de validação, limites duros e o protocolo do multi-agente com Recepcionista.

02 · O instalador

Do npm ao seu projeto em um comando

O CLI lê o manifest.json e busca cada arquivo direto do GitHub. O mesmo comando serve para instalar e para atualizar: arquivos existentes são sobrescritos, arquivos marcados como deprecated no manifesto são removidos.

1

Remove os legados

Arquivos listados em deprecated_files são apagados do projeto, com defesa contra path traversal: nada fora do diretório atual é tocado.

2

Baixa cada arquivo do manifesto

Um a um, da branch main do repositório oficial. Conteúdo idêntico ao que já está no disco é reportado como same e não é reescrito.

3

Grava e ajusta permissões

Cria as pastas que faltam, sobrescreve o que mudou e aplica chmod 755 nos scripts .sh dos hooks.

4

Resume o resultado

Contagem de adicionados, atualizados, sem mudanças, removidos e falhas. Qualquer falha de download encerra com código de erro.

03 · A arquitetura de agentes

Quatro papéis, uma regra de ouro

No atendimento em produção, o Orquestrador conversa com o lead e delega o resto: o Qualifier avalia, o Scheduler agenda e só o Protractor encerra sessões ou transfere para humano ou outro agente. Nenhum agente auxiliar fala com o lead diretamente.

Cada caixa é um template real de modelo/; as ações do Protractor são as cinco opções definidas no prompt dele.

Orquestrador

O único que fala com o lead no stack básico. Conduz a conversa em etapas numeradas (abertura, qualificação, pós-qualificação, agendamento) e aciona os auxiliares na hora certa.

Qualifier

Puramente analítico: recebe os dados coletados e devolve qualificado, desqualificado ou informações insuficientes, com justificativa. Não encerra conversas.

Scheduler

Interpreta linguagem natural de data e hora, converte para ISO e valida contra as regras reais de disponibilidade antes de marcar, remarcar ou cancelar.

Protractor

O sistema de decisão. Único autorizado a encerrar a sessão, transferir para humano ou para outro agente, pedir confirmação ao lead ou pausar follow-ups.

Follow-Up

Gera a mensagem de reengajamento personalizada pelo histórico da conversa — sem nunca expor a mecânica do follow-up para o lead.

Recepcionista multi-agente

Router opcional para clientes com várias frentes: identifica a intenção, transfere via Protractor e responde no mesmo turno personificando o especialista.

04 · Multi-agente

Quando um cliente tem várias frentes

Para clientes que atendem múltiplas áreas com fluxos distintos, o kit gera uma pasta Recepcionista/ na frente de stacks completos por especialidade. O lead nunca percebe a troca de agente.

No mesmo turno da transferência, o Recepcionista já envia a primeira mensagem personificando o especialista; a partir da resposta seguinte do lead, a especialidade assume de fato.

05 · O trabalho no dia a dia

Três slash commands dentro do Claude Code

Depois de instalado, todo o ciclo de vida de um cliente acontece por comando — criar, ajustar e atualizar — com subagentes especializados fazendo cada parte do trabalho.

/ei-cria-cliente <nome>

O subagente client-project-scaffolder copia os templates de modelo/ para a pasta do cliente, identifica os campos variáveis, coleta os dados com você e personaliza os arquivos. O que ficar sem resposta é marcado [PENDENTE].

/ei-ajustes <cliente> <descrição do problema>

Você descreve o problema em linguagem livre ("a IA está falando sobre valores") e o pipeline abaixo encontra, corrige e audita o prompt certo — com a sua aprovação no meio.

/ei-update

Re-executa o instalador na pasta atual e mostra o CHANGELOG da versão mais nova.

06 · O pipeline do /ei-ajustes

Analisar, aprovar, editar em paralelo, revisar em paralelo

O ajuste de um prompt em produção não é um "edita aí": é um pipeline com gate humano obrigatório, edição paralela e revisão cruzada, coordenado por um hook no evento Stop do Claude Code.

1

Localiza a pasta do cliente

Match exato, depois case-insensitive, depois substring. Se não existir, sugere /ei-cria-cliente.

2

docs-analyzer encontra o alvo

Subagente read-only lê os prompts do cliente e identifica, a partir da descrição livre, exatamente qual arquivo e qual seção precisam mudar.

3

Gate humano

A recomendação do analyzer é apresentada para aprovação explícita. Sem "Aprovar e editar", nenhum editor roda. Este passo não tem bypass.

4

Fan-out de editores

N instâncias paralelas de docs-editor-conciso, uma por arquivo aprovado. Cada uma emite OK ou ERRO; falhas têm retry isolado com teto de 2 tentativas.

5

Hook Stop encadeia a revisão

O script post-ajustes-fanout.sh detecta um sentinela no transcript e injeta a instrução de seguir para a revisão — protocolo stateless e idempotente, com fallback manual documentado.

6

Revisão cruzada em paralelo

M instâncias de docs-reviewer, cada uma enxergando as edições irmãs da rodada. Veredito OK, CORREÇÃO (re-edita, teto de 2) ou BLOQUEAR — e um resumo final consolidado.

07 · Guard rails

Limites que o sistema impõe a si mesmo

modelo/ é read-only

Os templates nunca são editados pelo fluxo de ajustes. Toda personalização acontece na cópia do cliente.

Aprovação humana obrigatória

Entre a análise e a edição existe um gate duro: sem resposta explícita, nenhum arquivo é alterado.

Casca XML validável

Todo template é envolvido numa raiz única <agente> com tipo e versão — validável com xmllint e byte-compatível com o Prompt Builder da SuperSDR.

Tetos de retry

Máximo de 2 tentativas por arquivo na edição e 2 ciclos de correção na revisão. Loop infinito não é opção — inclusive nos hooks, que usam sentinelas anti-reinjeção.

Defesa no instalador

Remoção de legados bloqueia path traversal: um manifesto comprometido não consegue apagar nada fora do diretório do projeto.

Auditoria sempre

Nenhuma edição termina sem passar pelo docs-reviewer, que confere formato de resposta, estrutura e consistência entre arquivos.
